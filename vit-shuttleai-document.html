System Design:

Architecture Overview:

VIT ShuttleAI follows a modern, modular, and lightweight static multi-page web architecture. The front-end is developed using Vanilla JavaScript with the Leaflet.js mapping library, eliminating framework overhead and ensuring maximum compatibility and performance across all browsers. The application is structured into six HTML pages, each loading a shared global stylesheet and shared JavaScript module, augmented by page-specific CSS and JS files.

The application follows a clear separation of concerns: the presentation layer is handled by HTML and CSS, the interaction layer is managed by page-specific JavaScript files, and shared cross-page functionality (clock, modal, scroll animations, navigation) is abstracted into a common shared.js module. This architecture ensures maintainability and enables seamless future integration with a Node.js backend or Firebase real-time database without requiring significant refactoring.

Bus position data is currently simulated using path-interpolation across pre-defined lat/lng waypoint arrays. Each bus has a progress counter that advances along its route array at 60fps using requestAnimationFrame(), and its Leaflet marker position is updated via setLatLng() on every frame. This architecture is designed so that actual GPS coordinates from a WebSocket feed can replace the simulated progress with minimal code change.

1. Set up the development environment: Install Node.js and npm on your machine. Clone or download the project directory containing all HTML, CSS, and JavaScript files. Ensure the folder structure maintains the /pages/ subdirectory for page-specific assets.

2. Understand the page architecture: Each HTML page (index.html, map.html, dashboard.html, schedules.html, alerts.html, ai-assistant.html) loads style.css and shared.js globally, then its own page-specific CSS and JS files. Changes to shared components propagate automatically across all pages.

3. Configure the map module (map.js): The Leaflet map is initialized with the Esri satellite tile layer, campus center coordinates (12.9692°N, 79.1559°E), and zoom level 16. Route waypoints are defined as lat/lng arrays calibrated to follow VIT campus roads visible on satellite imagery.

4. Manage bus animation: The animateBuses() loop runs via requestAnimationFrame(). Each bus object has a progress counter (0 to route.length-1), direction (1 or −1), and associated Leaflet marker. Bus positions are interpolated between adjacent waypoints using linear interpolation of lat/lng values. Driver data is injected into the marker icon HTML on every update frame.

5. Handle route filtering: Route filter buttons in the map overlay toggle the visibility of polylines and bus markers by adding/removing them from the Leaflet map instance. The "All Routes" button restores all layers simultaneously.

6. Manage the sign-in modal: The modal is dynamically injected into the DOM by shared.js at page load. The openSigninModal() function is exposed globally on the window object, allowing any navigation button across all pages to trigger it. Tab switching updates placeholder text and description to reflect student vs. admin context.

Lessons Learned:

Throughout the development process, we acquired invaluable lessons that will shape our future endeavours. We learned the importance of accurate geospatial calibration — early route waypoints that did not follow campus roads resulted in buses appearing to "jump" between buildings rather than travel on roads, which required careful re-calibration of all lat/lng arrays against satellite imagery. We also discovered that Leaflet.js CSS must be loaded in the HTML head rather than the body to ensure correct map tile rendering on page initialization.

Effective modularization of shared JavaScript functionality into a common shared.js module significantly reduced code duplication and made cross-page features (clock, modal, scroll animations) easy to maintain and update globally. We also found that CSS custom properties (design tokens) were invaluable for maintaining a consistent visual identity across six separate HTML pages without redundancy.

Furthermore, we prioritized animation performance by using requestAnimationFrame() exclusively for bus position updates rather than setInterval(), and by updating only the marker's setLatLng() method rather than re-rendering the entire DOM on every frame, ensuring consistently smooth 60fps performance even on lower-end devices.
